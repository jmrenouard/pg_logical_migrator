# Release v1.4.0

## [1.4.0] - 2026-05-05

### Features
- **migrator**: implement **UNLOGGED Tables synchronization**. Since PostgreSQL logical replication ignores `UNLOGGED` tables, this feature automatically identifies them, copies their data from the source, and syncs them to the target.
- **pipeline**: integrate `UNLOGGED` tables synchronization into the 17-step automated migration lifecycle.
- **wizard**: completed transition to a streamlined CLI-only interactive wizard architecture, completely removing the legacy `textual`-based TUI and its dependencies.

### Performance
- **migrator**: optimize Large Object (LOB) synchronization by introducing batch commits (every 100 objects), significantly reducing transaction overhead and improving data transfer speeds for databases with numerous LOBs.

### Bug Fixes
- **migrator**: remove string interpolation from parameterized SQL queries to prevent `TypeError` when logging statements (e.g. `check_sql`).
- **migrator**: fix `SyntaxError` in `sync_unlogged_tables` by properly implementing exception handling blocks.
- **pipeline**: fix `ValueError` when unpacking return values for databases with zero `UNLOGGED` tables. The function now consistently returns `(bool, str, int, list)` matching pipeline expectations.
- **migrator**: fix race condition in `wait_for_sync` by adding a retry loop for subscription visibility in `pg_subscription`.

### CI / CD & Packaging
- **packaging**: refactor `Makefile` and build scripts (`e2e_packaging_test.sh`) to centralize `PyInstaller` options via a shared `PYINSTALLER_OPTS` variable.
- **packaging**: removed legacy `textual` and `pytest-asyncio` dependencies to significantly reduce binary footprint and simplify the testing environment.
- **ci**: resolve persistent ActionLint warnings in `.github/workflows/pyinstaller-publish.yml` and `python-package.yml` by adjusting GitHub Actions versions (`v4`/`v5`) and fixing variable context evaluations (`env.VERSION`).
- **ci**: expand testing matrix and packaging support to explicitly include Python 3.11, 3.12, and 3.13 in all GitHub Actions pipelines.
- **ci**: updated `e2e_packaging_test.sh` to use `ubuntu:24.04` to ensure glibc consistency during binary validation.
- **ci**: integrated `pytest-cov` and HTML report generation explicitly into the Makefile (`test-coverage` target) and GitHub Actions.
- **ci**: automated test coverage artifact uploads for `python-package.yml` to preserve CI run metrics and reports.
- **ci**: hardened CI environment by ensuring proper cleanup of stale replication objects between test runs.

### Tests
- **tests**: add comprehensive unit tests (`tests/unit/test_migrator_unlogged.py`) to validate success, failure, and no-data scenarios for `UNLOGGED` table sync.
- **tests**: mock new `sync_unlogged_tables` method in `test_cli_pipelines.py` to maintain 100% test coverage and pipeline reliability.
- **tests**: successful End-to-End (E2E) verification of the new feature inside live Dockerized environments.
- **tests**: hardened E2E test `test_compiled_binary.py` against GLIBC version mismatch by intelligently skipping backwards compatibility tests on modern build hosts.
- **tests**: increased replication `wait_for_sync` timeout to 180 seconds to allow successful logical replication catch-up in heavily loaded or LOB-intensive integration tests.
- **tests**: refactored integration tests (`test_wizard_config.py`, `test_wizard_commands.py`) to properly mock standard input for the new CLI-only wizard.
- **tests**: refactored `test_step08_cleanup` to target specific publication and subscription names, preventing false negatives in dynamic environments.

### Configuration & Multi-Database 
- **config**: overhauled `Config.ini` parsing architecture to allow section-based per-database overrides (e.g. `[database:mydb]`), enabling true multi-tenant connection configuration.
- **migrator**: implemented safe schema hashing or generic suffixes when creating multi-schema logical replication slots to prevent PostgreSQL name-length errors.
- **migrator**: robust integration of multiple schema targets, correctly resolving database-specific variables like `target_schema` and `dest_host_for_src` across the pipeline.
